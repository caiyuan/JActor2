What is JActor2?

JActor2 is Java Lock Free Concurrent Framework

Why do we require Concurrent Programming?

The Moore's law(http://en.wikipedia.org/wiki/Moore%27s_law) is not longer valid and the hardware vendors have started putting more CPU cores to boost the processor speed. Writing application to take advantage of the multi core system require to write concurrent application. Here is the basics about the concurrency http://en.wikipedia.org/wiki/Concurrent_computing

Where do we require Concurrent Programming?

The Moore's law is no longer working and therefore hardware vendors are pushing for multicore processors, read this for more details http://www.gotw.ca/publications/concurrency-ddj.htm We need Concurrent Programming to leverage the compute utilization from multicore machines. Imagine of deploying the application on machine with 10's and 100's of cores, will the application perform well, will it scale etc.... We need to design/write Concurrent application to address this issue.

How do we write a traditional Concurrent Programming?

Typical concurrent applications are written using multi threads which prone to deadlocks if not written properly, JActor provides an alternate means to write an Concurrent applications which are easy to maintain.We will see it down.

What issue do we get writing a Concurrent Programs in a traditional way?

Hard to Write Correctly. Harder to Maintain.

Solving a major issue

Vertical scaling is today a major issue when writing server code. Threads and locks are the traditional approach to making full utilization of fat (multi-core) computers, but the result is code that is difficult to maintain and which to often does not run much faster than single-threaded code.

Actors make good use of fat computers but tend to be slow as messages are passed between threads. Attempts to optimize actor-based programs results in actors with multiple concerns (loss of modularity) and lots of spaghetti code.

The approach used by JActor is to minimize the messages passed between threads by executing the messages sent to idle actors on the same thread used by the actor which sent the message. Message buffering is used when messages must be sent between threads, and two-way messaging is used for implicit flow control. The result is an approach that is easy to maintain and which, with a bit of care to the architecture, provides extremely high rates of throughput.
